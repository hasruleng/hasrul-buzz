## Context

The game's *design* is fully decided — the challenge mechanic ([ADR-0007](adr-0007.md)), the scene flow
([ADR-0008](adr-0008.md)), and the metrics ([ADR-0010](adr-0010.md)). All three deferred one thing: the
**runtime** that draws and runs the world. This ADR decides it. (ROADMAP candidates: Phaser 3, Kaplay,
or a DOM/React tile renderer.)

[ADR-0008](adr-0008.md) pinned the bar precisely. The world needs: a **walkable avatar**, a small
**hub** with enterable **buildings**, **NPC dialogue**, an **exhibit view** (image + caption), a
**challenge overlay** (predict-then-reveal), a **teleport** (the magic door), and a small **journey
tracker** ([ADR-0010](adr-0010.md)). It explicitly noted this is "modest, top-down, **non-physics**" —
and [ADR-0007](adr-0007.md) said to "judge against a *modest* runtime need, not a AAA one."

That framing matters, because the usual reason to reach for a game engine — physics, real-time
simulation, sprite-heavy scenes, a tile/collision system — **none of it applies here**. The world is a
tiny graph of ~7 role-buildings plus a hub. "Movement" is walking between a handful of locations;
"collision" is staying in bounds and detecting which zone you're in. This is the regime where an engine
is overhead, not leverage.

Three of the repo's standing commitments push hard on this decision:

- **Stack discipline** (CLAUDE.md, [ADR-0002](adr-0002.md)): *"Do not add another framework/language
  without a new ADR — fighting stack-sprawl is a stated goal."* Phaser and Kaplay are each a new runtime
  dependency and a new mental model bolted onto a React app.
- **Content-first + the value is text** ([ADR-0001](adr-0001.md), [ADR-0011](adr-0011.md)): the entire
  product is *legible, accessible career content*. Canvas-based engines (both Phaser and Kaplay)
  rasterise text — it is **not selectable, not in the accessibility tree, and hard to keep at AA
  contrast**. For a site whose magic door is literally "skip to the readable text," rendering that text
  to a canvas is fighting our own grain.
- **Product judgment, not engineering-wow** ([ADR-0001](adr-0001.md), [ROADMAP.md](../ROADMAP.md)):
  *"Spend effort on the router and the challenge mechanic, not on graphics-wow."* An engine invites
  graphics scope-creep; staying in DOM keeps the effort on the mechanic.

## Prior art

An architecture decision; references are 2D-web-game implementation patterns.

**2D games built in pure DOM/React/SVG** (turn-based and grid/RPG-style games rendered with positioned
DOM nodes and CSS transforms, no canvas). *Borrowed:* the proof that a **non-physics, low-actor** 2D
world runs perfectly well as DOM — movement is state + `transform: translate`, "tiles" are a CSS grid or
absolutely-positioned elements. *Fits us because* the world is tiny and turnless.

**Phaser-in-React integration** (mounting a Phaser `Game` inside a React component via a ref). *Studied,
not adopted:* it is the canonical "engine in React," but it creates **two state worlds** — React owns the
content/router, Phaser owns the canvas — that must be manually bridged, and the canvas is opaque to the
DOM (a11y, deep links, text selection). The friction is the cost we avoid.

**Kaplay / Kaboom.js** (a lightweight, friendly 2D engine). *Studied, not adopted:* genuinely small and
pleasant, but it is still a canvas runtime with the same text-in-canvas accessibility cost and still a
new dependency + mental model for a mechanic this light.

**Internal prior art — we already render the map in DOM.** The [ADR-0011](adr-0011.md) visual specimen
(parchment, contour grid, station pins, a dashed trail, a hub mock with buildings and a vista) is
**entirely HTML/CSS/SVG**, and the readable timeline already renders as a route. The cartographic world
is proven to work as DOM today; a DOM game *extends* this, while a canvas game *re-implements* it.

**Accessibility guidance on canvas text** (WCAG / MDN: canvas content is not exposed to assistive tech
unless laboriously mirrored in fallback DOM). *Borrowed as a constraint:* keep meaningful text in real
DOM.

**What we did *not* borrow — and won't.** Heavy 2D physics engines (Matter.js et al. — no physics need);
3D / WebGL spectacle (Three.js / R3F — rejected in [ADR-0001](adr-0001.md), reaffirmed); and adopting a
named engine *for résumé optics* rather than fit — that is engineering-wow, the exact axis
[ADR-0001](adr-0001.md) rejects (but see Alternatives for the honest counter-argument).

## Decision

**Render the Phase-3 world in DOM/React/SVG/CSS — no game engine.** Concretely:

- **Avatar** — a positioned DOM/SVG element moved with `transform: translate`, driven by React state.
- **Hub & buildings** — laid out as DOM/SVG using the [ADR-0011](adr-0011.md) cartographic tokens
  (parchment, contour grid, pins, trail, the vista). The map *is* styled markup, as in the specimen.
- **Movement & zones** — keyboard + touch handlers update a small "world state" hook; no physics, no
  tilemap collision — just position bounds and "which building am I near / in."
- **Dialogue, exhibit, challenge, summary** — plain **React components** that reuse the existing content
  model ([ADR-0003](adr-0003.md)) and persona framings directly. The challenge overlay is the
  [ADR-0007](adr-0007.md) predict-then-reveal UI; the exhibit shows a real screenshot + caption.
- **Magic door & deep links** — the **existing router**: the magic door is a navigation to the finale /
  readable content, and the owed `&station=` param ([ADR-0008](adr-0008.md)) makes any building
  directly linkable. No bridge between two state worlds — there is only one (React).
- **Journey tracker** ([ADR-0010](adr-0010.md)) — a small client-side hook; trivially in-React.

**This adds no new framework or language** — it is the [ADR-0002](adr-0002.md) stack, unchanged. The one
honest cost is that we hand-write avatar movement and simple zone logic instead of getting them free from
an engine. Given a ~8-location, non-physics world, that is a few hundred lines, not an engine's worth —
and it buys us real DOM (accessible, selectable, deep-linkable, AA-legible), one state world, a tiny
bundle, and zero new dependency.

## Alternatives considered

- **Phaser 3** — the mature default: tilemaps, sprites, input, tweens, scenes for free. But it is a heavy
  canvas runtime (large bundle — slow for the hurried recruiter), its **text is invisible to a11y and
  hard to keep legible**, it forces a **two-state-world bridge** with React, and all that power targets
  physics/real-time needs **we do not have**. *Rejected as the default — overkill that fights
  content-first.*
- **Kaplay / Kaboom** — much lighter and friendlier than Phaser, a real contender on bundle and DX. Still
  a **canvas** runtime (same text-a11y/legibility cost) and still **a new dependency + mental model** for
  a mechanic this light. *Rejected on content-first / stack-discipline grounds — the DOM renderer gets us
  there without a new runtime.*
- **Adopt a named engine as a portfolio skill-flex** (so a recruiter sees "shipped with Phaser/Kaplay").
  This is the one argument the repo's principles *don't* capture, and it is legitimate — but it cuts
  directly against the north star (*product judgment, not engineering-wow*) and stack discipline. *Flagged
  for Hasrul to weigh explicitly; if he wants the engine itself as a demonstrable skill, that — not
  runtime need — would be the reason to revisit, and it would be recorded as a superseding ADR.*
- **3D / WebGL (Three.js / React-Three-Fiber)** — highest ceiling, wrong axis; rejected in
  [ADR-0001](adr-0001.md). *Reaffirmed.*

## Consequences

- **No new runtime dependency; the bundle stays small** — fast load for the time-pressed recruiter, in
  keeping with ship-fast and content-first.
- **A small "world" layer is owed in Phase 3** — a world-state hook (avatar position, current zone,
  unlocked/visited set), movement handlers (keyboard + touch), and the hub/building DOM-SVG. Modest.
- **The content components are reused, not re-built** — dialogue/exhibit/challenge/summary are React over
  the [ADR-0003](adr-0003.md) model; the [ADR-0011](adr-0011.md) tokens already exist.
- **One state world** — the router, persona state, deep links, and the world all live in React; the magic
  door and `&station=` deep link ([ADR-0008](adr-0008.md)) are plain navigation, not a canvas bridge.
- **Accessibility and legibility are preserved by construction** — every word stays real DOM.
- **[ADR-0002](adr-0002.md) stays intact** — no framework added; stack-sprawl avoided.
- **Supersede trigger:** if the game ever genuinely needs real-time simulation, physics, or many animated
  actors beyond what DOM handles smoothly, revisit with a new ADR — though that same need would signal
  scope drifting toward the graphics-wow the north star resists. Or, if Hasrul decides the engine is a
  portfolio goal in itself, a superseding ADR records that trade.
