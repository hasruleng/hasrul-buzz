# ADR-0002: Frontend stack — Vite + React + TypeScript (SPA-first, SSG-later)

← [ADR index](../ADR.md)

**Status:** Accepted (2026-06-20)
**Drivers (ranked by Hasrul):** ① ship fast · ② align to existing stack / stop stack-sprawl ·
③ flexibility to meet the [ROADMAP](../ROADMAP.md). **SEO is an explicit non-goal** (see Context).
**Constrains / constrained by:** [ADR-0001](adr-0001.md) (content-first, game-optional)

## Context

[ADR-0001](adr-0001.md) commits to a content-first, game-optional site with three layers. The
frontend stack must serve four things that pull in different directions:

- **Ship fast** on a weekend cadence — minimal config, fast first deploy.
- **Reuse existing skills** — Hasrul ships React and C# .NET; a React-friendly path lowers friction.
- **Grow into a game** — Phase 3 embeds a Phaser/Kaplay pixel-walker as one heavy client-side island.

SEO / discoverability is **not** a constraint on this choice — it is an explicit non-goal (see criteria
below).

**Hasrul's ranked criteria (stated 2026-06-20):**

1. **Ship fast** — the overriding priority. Minimise new concepts and config; deploy quickly; iterate
   every weekend.
2. **Align to the existing stack / stop stack-sprawl** — "I already have too many stacks on my belt,
   I need to focus." The stack should *reuse* what Hasrul already ships (React), not add another.
3. **Flexibility to meet the [ROADMAP](../ROADMAP.md)** — persona-router → content site → map/skill-tree
   → Phaser game → (optional later) content shell.

**SEO is explicitly NOT a criterion.** Hasrul: *"I don't care about SEO. Most of my visitors will be
recruiters/founders who already have my application (CV / LinkedIn profile). Right now I don't need to
be a famous Product Engineer — I just want to land a job."* Traffic is **direct-link, not search**.
This removes the single weakness of a client-rendered SPA and is the deciding factor below.

## Prior art

**Islands architecture.** The "mostly-static HTML with small interactive islands" pattern was named by
[Katie Sylor-Miller / Jason Miller, "Islands Architecture" (2020)](https://jasonformat.com/islands-architecture/)
and is Astro's core model ([Astro docs — Islands](https://docs.astro.build/en/concepts/islands/)).
*Fit:* it matches [ADR-0001](adr-0001.md)'s split exactly — readable persona pages are static/indexable;
the persona-router and the Phase-3 game are the interactive islands.

**Content-first site with a separate game route.** hajj.buzz (see [ADR-0001](adr-0001.md) prior art)
ships a readable content site plus the game at `/app` — the production proof that this split works for
a real audience. *Fit:* validates serving the game as an opt-in route rather than the whole site.

**Forkable game scaffolds.** The Phase-3 engine choice will lean on existing 2D pixel-resume repos
([petejank/interactive-resume](https://github.com/petejank/interactive-resume),
[dokson/interactive-resume](https://github.com/dokson/interactive-resume)) and the mature
[Phaser 3](https://phaser.io) / [Kaplay](https://kaplayjs.com) engines — so "grows into a game" is a
known path, not speculation.

*How the criteria resolved this:* the islands/SSG lineage is a real strength **only if SEO matters** —
and it doesn't (see Context). With SEO off the table, the static-HTML advantage of Astro stops being
decisive, and criteria ① and ② (ship fast, stay in React) win the day.

## Decision

**Vite + React + TypeScript**, deployed as a single-page app. Persona state lives in URL query params
(`?persona=&lens=`) — deep-linkable, no state library. The Phase-3 Phaser/Kaplay game mounts as a
Phaser canvas inside a React route (`/play`); Phaser is framework-agnostic and drops into any `<div>`.

**SPA-first, SSG-later:** ship as a client-rendered SPA now. *If* SEO ever becomes a goal (it isn't
today), add static prerendering to the **same** app with [`vite-react-ssg`](https://github.com/Daydreamer-riri/vite-react-ssg)
or [Vike](https://vike.dev) — an additive plugin over the existing components, not a migration.

Why this beats the alternatives on Hasrul's ranked criteria:

- **① Ship fast** — Vite + React is Hasrul's existing stack; near-zero new concepts, minimal config,
  first deploy in ~an hour. Fastest of all options.
- **② Stack focus** — pure React, no new framework and no new language. *Reduces* stack count rather
  than adding Astro/Svelte to the belt.
- **③ Roadmap flexibility** — every phase is native React: router = query params + components;
  map/skill-tree = React + SVG/animation; game = Phaser canvas in a route. The only roadmap item that
  would have needed more (SEO content shell, Phase 4) is a non-goal, and is an additive plugin if it
  ever changes.

## Alternatives considered

| Option | Strengths | Weaknesses |
|---|---|---|
| **Vite + React + TS (SPA)** *(chosen)* | Zero learning curve; fastest to first pixel; Hasrul's existing stack (criteria ① + ②); ideal host for the game; SSG can be added later as a plugin if ever needed. | Client-rendered → weak SEO/link-previews — **irrelevant here, SEO is a non-goal**. |
| **Astro + React islands** | Static HTML by default → strong SEO, skimmable; islands for router + game; maps onto the hajj.buzz content-vs-/app shape. | Its main edge (SEO) is moot here; a *new* framework + islands model works against criterion ② (stack focus). |
| **Next.js (React)** | Known language (React); full SSG/SSR → excellent SEO; one framework for everything. | Heavier; App-Router conventions = more week-1 plumbing → loses on criterion ① (ship fast); SEO edge is moot. |
| **SvelteKit** | Excellent DX and runtime performance; great SEO. | New *language* (Svelte) fights ① and ②; React skills don't transfer. |

## Consequences

- Commits to a client-rendered React SPA; everything stays in one familiar language and toolchain.
- **No SEO / link-preview / no-JS support out of the box.** Accepted deliberately — traffic is
  direct-link from CV/LinkedIn, not search. Revisit only if that assumption changes.
- The Phase-3 game mounts in a React route, not a rewrite — preserves "no wasted work."
- Persona deep-links work natively via URL query params; no router or state library is strictly
  required to start (can add React Router later if routes multiply).
- Downstream decisions **not** settled here: game engine (future ADR-0003), hosting (ADR-0004),
  styling (ADR-0007).
- **Supersede trigger:** if SEO/discoverability ever becomes a real goal and the `vite-react-ssg`/Vike
  add-on proves insufficient, a move to Astro or Next.js would be the likely path — recorded as a *new*
  ADR superseding this one, never an in-place edit.
