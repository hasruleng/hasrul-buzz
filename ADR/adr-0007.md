# ADR-0007: Challenge mechanic — per-station "evidence checks" that make each persona perform its own evaluation

← [ADR index](../ADR.md)

**Status:** Accepted (2026-06-21)
**Owed by:** [ADR-0005](adr-0005.md) ("a new decision is owed: what the challenges *are* … before any game code is written")
**Extends:** [ADR-0001](adr-0001.md) (content-first / skippable still bind) · [ADR-0003](adr-0003.md) (reuses the content model; adds a thin layer)
**Drivers:** persona-specific discovery · product judgment over graphics-wow · memorability · honesty (§A/§E)
**Feeds:** [ROADMAP.md](../ROADMAP.md) Phase 3 · **does not** decide the game engine (that is a separate owed ADR)

## Context

[ADR-0005](adr-0005.md) made gamified discovery a co-differentiator and committed the site to a game
that **gates advancement behind challenges layered on top of walk-and-talk** — walk-and-talk is the
base; challenges are what push past the walk-and-talk-only approach and differentiate from hajj.buzz.
It also ruled out generic trivia. It then deferred the actual question: **what is a challenge?** No
game code may be written until that is decided. This ADR decides it. (It does **not** choose the
engine — Phaser vs Kaplay vs a DOM renderer — which remains an owed, separate decision.)

The constraint that shapes everything: the one goal ([ADR-0005](adr-0005.md)) is to help *each persona
discover the specific things about Hasrul that matter to their interest.* So a challenge cannot be a
decorative quiz bolted onto the side. The challenge **must be the discovery** — passing it should mean
the visitor has engaged the exact fact their persona came for. A trivia question ("what year did he
graduate?") fails this test: it gates progress without surfacing anything role-relevant. The mechanic
has to make the *recruiter* do recruiter-thinking, the *founder* do ownership-thinking, the *casual*
visitor meet the human.

A second hard constraint is inherited and non-negotiable: **content-first and skippable**
([ADR-0001](adr-0001.md)). The readable site holds every fact unconditionally; the game is always
skippable. So a challenge can never be a *hard gate* that withholds a fact until you answer correctly —
that would punish the hurried recruiter and break the canonical-site guarantee.

## Prior art

**hajj.buzz — challenge-gated station progression** (the model named in [ADR-0005](adr-0005.md)).
*Borrowed:* the challenge-to-advance loop — you complete something at a station to move on. *Differs:*
hajj.buzz asks the same fixed questions of everyone; here the check is **persona-specific** and is an
*evaluation/matching task*, not recall, and a wrong answer reveals-and-teaches rather than blocking.

**NYT "You Draw It" / predict-then-reveal interactives.** Readers commit a guess (drawing a trend)
*before* seeing the real data; the act of committing makes the reveal land and stick. *Borrowed:* the
**predict-then-reveal** device as the core engagement + memorability mechanism — directly serving the
"be remembered" goal. *Differs:* our "data" is persona-framed career evidence, not public statistics.

**Duolingo — low-stakes checks.** Questions gate lesson progress, but a wrong answer reveals the right
one and you continue; there is no punishment, only teaching. *Borrowed:* **reveal-and-teach, never
hard-fail** — the loop nudges engagement without withholding anything. *Differs:* we surface one
person's verified facts to a self-selected audience, not a teaching curriculum.

**Structured recruiter screening (requirement → evidence matching).** A recruiter's actual job is
mapping a role requirement to proof in a candidate's history. *Borrowed:* making the recruiter-persona
check *be that task* — so the mechanic mirrors the visitor's own work and feels meaningful rather than
arbitrary. *Differs:* lifted into a playful, self-serve form, and the "candidate" is fixed.

**RPG dialogue / skill checks.** In-world, a check is framed as part of the narrative (an NPC asks; a
choice advances the scene). *Borrowed:* framing each check as station/NPC dialogue so it reads as story,
not a form. *Not adopted:* pass/fail punishment, stat-RNG, or "game over" states.

**What we did *not* borrow — and won't.** Generic trivia quizzes (recall of dates/names — rejected by
[ADR-0005](adr-0005.md) by name); leaderboards / competitive scoring / timers (wrong axis — this is
discovery, not competition, and timers punish the hurried); and hard-fail gates that withhold content
(break [ADR-0001](adr-0001.md)'s content-first guarantee).

## Decision

The challenge is a **per-station "evidence check": a short, persona-framed decision the visitor commits
to, which then reveals the role-relevant fact and unlocks the next station.** The visitor performs *their
own persona's evaluation* of Hasrul; the act of committing to an answer is what makes the revealed fact
memorable (predict-then-reveal). One check per station.

**The loop (per station):**

```
   ┌─ PROMPT ──────────────────────────────────────────────┐
   │ A question framed by the active persona / lens. It can │  ← the router (ADR-0005 D1)
   │ only be answered by engaging a role-relevant fact.     │     chooses which framing
   └────────────────────────────────────────────────────────┘
                       │ visitor commits a choice (2–4 options / a match)
                       ▼
   ┌─ REVEAL ──────────────────────────────────────────────┐
   │ The verified fact + its proof (project, metric, story) │  ← the discovery moment
   │ from the content model, framed for the persona.        │     (predict-then-reveal)
   │ Wrong pick → reveal-and-teach, never punish.           │
   └────────────────────────────────────────────────────────┘
                       │ station unlocks
                       ▼
              avatar walks on → next station
```

**Three persona framings** (illustrative, drawn entirely from facts already published on the readable
site — see [stations.ts](../src/content/stations.ts) / [lenses.ts](../src/content/lenses.ts)):

- **Recruiter (per lens) — gamify the recruiter's own job: requirement → evidence.**
  *Prompt:* "You're hiring a **backend** engineer for a high-throughput real-time system. Which of
  Hasrul's projects is the strongest proof?" *Options (all real, all his):* Transjakarta distributed
  real-time GPS · EV Fleet CAN-bus ETL · PINTAR national platform. *Reveal:* Transjakarta — a custom
  distributed real-time GPS for 1M+ daily commuters and 7,000+ buses. The check *is* a fit assessment.

- **Founder / Owner — an ownership scenario.**
  *Prompt:* "Founder test: which of these did **one engineer own end-to-end** — UI, backend, cloud, and
  infra?" *Options:* EV Fleet Dashboard · plus two genuine but team-built projects. *Reveal:* EV Fleet —
  React + Laravel + AWS/Athena + CAN ETL + Terraform, architected and delivered solo (with the honest
  note that the others were team efforts).

- **Casual / Stalker — the human / origin.** (A lighter, more trivia-like touch is *on-brand here*,
  because for this persona "getting to know the human" is the goal.)
  *Prompt:* "Before the engineering — what did 19-year-old Hasrul run?" *Reveal:* a dorm internet café
  (Warnet) he ran for ~8 years from his 2nd undergrad semester — the origin story.

**Honesty is a design constraint of the mechanic, not an afterthought (§A/§E):**

- Every prompt, option, and reveal traces to a verified fact in the content model. **No distractor may
  assert a false claim about Hasrul.** Safe distractor patterns: (a) *all options are real items of his*
  and the task is correct association; (b) options are well-known generic items clearly not claimed as
  his; (c) clearly-hypothetical framing. Never "he built «fabricated thing»".
- The reveal always states the verified truth, so a wrong pick is corrected immediately — the visitor
  never leaves with a false impression.
- §E gaps are fair game as honest checks (e.g. a lens whose reveal names the stack-deficit + the bridge),
  never hidden.

**It reuses the content model; it does not add truth ([ADR-0003](adr-0003.md)).** A challenge is a thin
additive layer — a `challenge` derived per station (with per-lens variants where the framing differs) —
built from the *existing* verified facts and persona framings. New renderer, not new content. Exact
schema is deferred to implementation, but the rule binds: challenges may only reframe facts already in
the model.

**Visitor context shapes the challenge goal — especially for recruiters.** Most recruiters arrive
having already read a tailored CV (they received it with a job application). They come to the site
with context, not without it. A smaller segment arrives cold — they found a LinkedIn profile and
clicked through. These two segments need different things from the challenge:

| Recruiter segment | What they come with | What the challenge should do |
|---|---|---|
| **CV-equipped** (majority) | Already knows the facts | Surface depth the CV cannot: the full story behind a bullet point, the context, the human detail |
| **LinkedIn-cold** (minority) | No prior context | Introduce the evidence and let them reach the conclusion themselves |

Founders arriving from LinkedIn are similar to the LinkedIn-cold recruiter — cold entry, discovery
mode. Casual visitors are always cold by definition.

**Each challenge must be evaluated against the emotional arc of its persona:**

| Persona | Desired emotion | Failure mode to avoid |
|---|---|---|
| Recruiter (CV-equipped) | **Recognition + depth** — *"I knew this from the CV, but now I understand the full picture"* | Redundancy — replaying what the CV already said without adding anything |
| Recruiter (LinkedIn-cold) | **Confidence** — *"I reached that conclusion myself"* | Annoyance — feels like a quiz on Hasrul's behalf, not a tool for the recruiter |
| Founder | **Conviction** (if confirmed) or **genuine discovery** (if surprised) — both positive | Condescension — framing that makes the founder feel tested rather than informed |
| Casual | **Delight** — a good story surfaced at the right moment | Indifference — a challenge so generic it could be about anyone |

A challenge only creates a positive emotion if it feels like it's **for the visitor** (helping them reach their own conclusion or see deeper) rather than **for Hasrul** (a test they must pass to continue).

**A "magic door" shortcut is always available within the game.** A visitor can jump directly to a
specific station without walking through every prior station and challenge. This goes beyond the "skip the game entirely" escape hatch
([ADR-0001](adr-0001.md)): even *within* the game, no visitor is ever trapped in the walk-and-talk
sequence. The station map / index is always one action away, and deep links into specific stations
are supported (extending the `?persona=&lens=` deep-link model to include a `&station=` parameter).
This is especially important for recruiters who arrive with a specific question and need the answer
in seconds, not after walking through three prior stations.

**It stays skippable and content-first ([ADR-0001](adr-0001.md)).** Checks are *reveal-and-continue*,
never hard gates: a wrong (or skipped) answer still reveals the fact and unlocks the station. The
readable site continues to hold every fact unconditionally, and "skip the game" is always one action away.

## Alternatives considered

- **Walk-and-talk only (no challenge)** — walk-and-talk is the base and is kept; adding *no* challenge
  on top was rejected by [ADR-0005](adr-0005.md) as too thin. No commitment step → no predict-then-reveal
  stickiness → visitors walk through and forget.
- **Generic trivia quiz** (recall of dates/names) — gates progress without surfacing anything
  role-relevant. *Rejected — the exact anti-pattern [ADR-0005](adr-0005.md) named.*
- **Hard skill-check gating** (must answer correctly to advance) — adds stakes, but withholds canonical
  facts, punishes, and loses the hurried recruiter. *Rejected — breaks content-first.*
- **One end-of-journey quiz** — simpler to build, but detaches the check from the moment of discovery;
  per-station keeps the prediction adjacent to its reveal, which is where the memory effect lives.
  *Rejected.*
- **Free-text / open answers** — richest in theory, but fragile to evaluate, high-friction, and risks
  false "fails" on correct-but-differently-worded answers. *Rejected.*
- **Branching dialogue trees with no check** — engaging, but it is walk-and-talk by another name; no
  commitment step, so no predict-then-reveal. *Rejected.*

## Consequences

- **A thin additive content layer is owed at implementation:** a `challenge` per station (+ per-lens
  variants), derived from existing facts. It must add no new truth; §A/§E bind every prompt/option/reveal
  and **forbid false-claim distractors**. The exact schema lands with the Phase-3 code, reusing
  [ADR-0003](adr-0003.md)'s model.
- **This informs the engine-choice ADR (next, separate).** The loop is light: dialog/choice UI + a
  walking avatar + simple unlocked-station state — not physics-heavy. So the engine decision (Phaser vs
  Kaplay vs even a DOM/React tile renderer) can be judged against a *modest* runtime need, not a AAA one.
- **The persona-router still chooses WHAT; the challenge renders it** — the same station poses a
  different check per persona/lens (the [ADR-0005](adr-0005.md) D1→D2 relationship, made concrete).
- **A `&station=` deep-link parameter is owed** — extends the existing `?persona=&lens=` model so
  any station is directly linkable from outside the game. The "pintu kemana saja" shortcut within the
  game is its in-session equivalent.
- **The recruiter fit-meter** ([ROADMAP.md](../ROADMAP.md) Phase 4) gets a natural feeder: each check
  passed advances discovery; the meter can fill as role-relevant evidence is surfaced. Optional.
- **Skippable / content-first preserved** — reveal-and-continue, never a hard gate.
- **Supersede trigger:** if Phase-4 analytics show challenge-players don't recall more role-relevant
  facts (or convert no better) than plain readers, a future ADR may simplify the mechanic or demote it —
  recorded as a new ADR, never an in-place edit here.
